Design · design

/twt-design

Run the full design phase — system, components, layouts, mockups — into one handoff-ready brief.

Version 1.3.7 Arguments: yes

Purpose

Runs your whole design phase — design system, components, page layouts, and mockups — and ties it all into one brief that's ready to hand to development. It's the quickest way to get from a pre-design brief to a design you can actually build.

Skills it runs, in order

This skill is an orchestrator — it dispatches the skills below (via the Agent tool) rather than doing their work itself. Here's each one, in the order it runs, and why.

  1. /twt-design-systemEstablishes tokens and components — the shared spine every page is built on.
  2. /twt-component-define or /twt-component-validateSpecs each UI component, then checks it for reuse, state coverage, and accessibility.
  3. /twt-layout-define or /twt-layout-validateSpecs each page's layout, then reviews hierarchy and responsive intent.
  4. /twt-mockup-define or /twt-mockup-validateRenders real-content mockups, then checks them for fidelity and accessibility.

Parameters it can expect

Optional design sources; optional --from/--only flags (area ∈ design-system/component/layout/mockup)

Result it provides

  • .twt-artifacts/design/design-brief.mdThe Phase-2 handoff brief — design system, components, layouts, and mockups tied together for development.
  • .twt-artifacts/design/design-read.mdThe agreed visual direction and design 'dials' that steer the whole design phase.
  • .twt-artifacts/design/decisions.mdA log of the key decisions and trade-offs made while producing this artifact.

Boundaries

  • Doesn't do development or QA (later phases)
  • Doesn't reproduce sub-area logic — dispatches every sub-area's *-define / *-validate sub-skills directly (rule 5); the standalone /twt-design-system wrapper is for direct use, not for routing through (an orchestrator-to-orchestrator hop only relays)
  • The brief is a static synthesis, not a live transition skill

Success criteria

  • Each requested sub-area runs in order: design-system → component → layout → mockup
  • --from <area> resumes from a sub-area; --only <area> scopes to one
  • design-brief.md summarizes the system, components, layouts, and mockups with links to every artifact, and surfaces any outstanding BLOCKERs
