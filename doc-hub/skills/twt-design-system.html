Design · design-system

/twt-design-system

Define or analyze your design system and validate it in a single pass — the shared spine for every phase.

Version 1.3.2 Arguments: yes

Purpose

Sets up the foundation every page stands on — colors, type, spacing, and components — either from scratch or by studying designs you already have, then checks it for consistency. This is the shared reference that design, development, and QA all point back to.

Skills it runs, in order

This skill is an orchestrator — it dispatches the skills below (via the Agent tool) rather than doing their work itself. Here's each one, in the order it runs, and why.

  1. twt-design-system-defineBuilds or analyzes the tokens, components, and preview.
  2. twt-design-system-validateRuns a read-only critique, including a WCAG contrast gate.

Parameters it can expect

Optional design sources (Figma/screenshots/URL) or none (greenfield from brand-brief)

What it reads

  • .twt-artifacts/design/design-system/tokens.mdThe human-readable token reference — colors, type, spacing, and more.
  • .twt-artifacts/design/design-system/validation-report.mdA read-only critique of this artifact, listing blockers, warnings, and suggestions.

Result it provides

  • .twt-artifacts/design/design-system/tokens.mdThe human-readable token reference — colors, type, spacing, and more.
  • .twt-artifacts/design/design-system/tokens.cssThe design tokens as CSS custom properties, ready for the build to consume.
  • .twt-artifacts/design/design-system/preview.htmlAn HTML page.
  • .twt-artifacts/design/design-system/decisions.mdA log of the key decisions and trade-offs made while producing this artifact.
  • .twt-artifacts/design/design-system/validation-report.mdA read-only critique of this artifact, listing blockers, warnings, and suggestions.
  • .twt-artifacts/design/design-system/component/components.mdA Markdown document.
  • .twt-artifacts/design/design-system/component/gallery.htmlAn HTML page.
  • .twt-artifacts/design/design-system/component/validation-report.mdA read-only critique of this artifact, listing blockers, warnings, and suggestions.

Boundaries

  • Doesn't reproduce sub-skill logic — dispatches via the Agent tool (rule 5)
  • Doesn't loop unbounded or auto-downgrade severity
  • Not required for standalone use — each sub-skill works on its own

Success criteria

  • Produces/refines tokens.md, tokens.css, a tokens-only preview.html (the component catalog lives in the gallery, linked from preview) and a current validation-report.md
  • Always builds the full component catalog (component/components.md + gallery.html) via /twt-component-define — in every mode, standalone or collect — and validates it (/twt-component-validate, or the collect-mode self-check) into component/validation-report.md. A complete design system includes tokens, preview, AND the full catalog of all primitives, components, and modules.
  • Honors the §9 single-pass policy: one define + one validate (folded into define under orchestration), at most one BLOCKER-driven re-run, no score-chasing loop; reports final Band + Health and surfaces open decisions per §13 (or bubbles them up in collect mode)
  • On exit, states final Band + Health and whether BLOCKERs remain
